# Upgrade the Java toolchain and runtime from Java 17 to Java 25 — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Move OAG from a Java 17 toolchain/runtime to Java 25 (LTS) across the Gradle build, the Docker build- and runtime-images, the CI/release workflows and the documentation, upgrading Gradle to its newest stable release along the way, and keeping the build fully green with no disabled or skipped tests.

**Architecture:** This is a toolchain-and-runtime version bump of the single `oag/` Gradle project — there is no new production code. Spring Boot 4.1 (already in use) supports Java 25, so no source changes are expected; if a Java-25 language/API change does surface, it is a genuine regression to fix, not to suppress. The safety net is the existing test suite — 43 test classes / 108 tests including full `@SpringBootTest` integration tests (WireMock, `WebTestClient`) that boot the complete Spring context and exercise gateway routing, CSRF, session handling, OIDC login and security config. Each task's "test" is therefore compiling and running that entire suite on Java 25 via `./gradlew build`. Because the current local JDK is 21 (not 25), the Gradle toolchain must be able to obtain a JDK 25 — an early task wires in auto-provisioning so the build is reproducible on any machine.

**Tech Stack:** Spring Cloud Gateway (server-webflux) · Spring Boot 4.1.0 · Spring Cloud 2025.1.2 (Oakwood) · Gradle (latest stable) · Java 25 toolchain (foojay auto-provisioned) · Docker

## Global Constraints

- **Java target = 25 exactly** (`JavaLanguageVersion.of(25)`), the current LTS. Do not target a non-LTS (26/27) and do not leave any Java-17 reference behind.
- **Gradle → newest stable release.** Upgrade the wrapper to the latest GA Gradle (discovered at execution time — see Task 1). Ignore any `-rc`, `-milestone`, `-M`, `-RC`, `-alpha`, `-beta` build. The chosen Gradle version must support a Java-25 toolchain (Task 2 verifies this); the latest GA does.
- **Do not change Spring Boot or Spring Cloud versions.** Spring Boot stays `4.1.0`, Spring Cloud `2025.1.2`. This iteration changes only the Java version and Gradle.
- **Baseline must be preserved:** before starting, `./gradlew build` prints `BUILD SUCCESSFUL` with all tests passing on the current Java 17 toolchain / Gradle 9.6.1. Every task ends with `BUILD SUCCESSFUL`. Never weaken, skip (`-PskipTests`), or `@Disabled` a test to make the bump "pass".
- **Both Docker stages move to Java 25** — the `gradle:*-jdk*` build stage and the `amazoncorretto:*` runtime stage. A build image on 25 with a runtime still on 17 would fail at container start (class-file version mismatch); keep them in lockstep. The build-stage `gradle:` tag must also match the upgraded wrapper version.
- **Verify image tags and plugin/JDK/Gradle versions at execution time.** Do not assume a Docker tag (`gradle:<ver>-jdk25`, `amazoncorretto:25-…`), a Gradle release, or a plugin version exists — confirm it with the command given in the step before pinning it.
- **Config rule:** one `spring:` block per YAML file (SnakeYAML merges duplicate top-level keys unpredictably). This upgrade should not add YAML, but honour it if any config is touched.
- **Language:** English in code, config, commit messages and docs. Documentation states the *current* state only — no "changed from Java 17" history in the docs themselves.
- **Commit after every task.** Frequent, small commits. Do not squash tasks together.

**Working directory:** all `./gradlew` commands run from `oag/`. `docker build` runs from the repository root. Paths below are given relative to the repository root.

- [x] **Step 1: Confirm the green baseline**

- [x] **Step 2: Discover the latest stable Gradle version**

Run: `curl -sSL https://services.gradle.org/versions/current`
Expected: a JSON blob whose `"version"` field is the latest GA (e.g. `"version" : "9.7.0"`). Record that version as `<GRADLE_VERSION>`. - [x] **Step 3: Fetch and record the distribution checksum (supply-chain hardening)**

Run (substituting the version from Step 2): `curl -sSL https://services.gradle.org/distributions/gradle-<GRADLE_VERSION>-bin.zip.sha256`

- [x] **Step 4: Run the wrapper upgrade task (two passes)**

Run (from `oag/`) twice, substituting both values — the two-pass procedure is required so the second pass regenerates `gradle-wrapper.jar`, `gradlew` and `gradlew.bat` with the new version:

```bash
./gradlew wrapper --gradle-version <GRADLE_VERSION> --distribution-type bin \
  --gradle-distribution-sha256-sum <CHECKSUM_FROM_STEP_3>
./gradlew wrapper --gradle-version <GRADLE_VERSION> --distribution-type bin \
  --gradle-distribution-sha256-sum <CHECKSUM_FROM_STEP_3>
```

- [x] **Step 5: Verify the wrapper version and properties**

Run (from `oag/`): `./gradlew --version`
Expected: the `Gradle` line reads `<GRADLE_VERSION>`.

Read `oag/gradle/wrapper/gradle-wrapper.properties` and confirm:
- `distributionUrl=https\://services.gradle.org/distributions/gradle-<GRADLE_VERSION>-bin.zip`
- `distributionSha256Sum=<CHECKSUM_FROM_STEP_3>`
- `validateDistributionUrl=true` is still present.

- [x] **Step 6: Full build on the new Gradle (still Java 17 toolchain)**

Run (from `oag/`): `./gradlew clean build --console=plain`
Expected: `BUILD SUCCESSFUL`, all tests pass. This isolates the Gradle bump from the Java bump — if it fails here, it is a Gradle-compatibility issue, not a Java-25 one.

If it fails with a "deprecation now removed" error, read the error's `--stacktrace`, find the offending line in `oag/build.gradle`, and replace it with the Gradle equivalent named in the message (re-run with `--warning-mode all` to surface the exact deprecation). Do not disable tasks or tests to get past a failure.

- [x] **Step 7: Commit**


### Task 2: Move the Gradle toolchain to Java 25 (with auto-provisioning)

**Files:**
- Modify: `oag/settings.gradle` (add the foojay toolchain-resolver plugin)
- Modify: `oag/build.gradle:37` (`JavaLanguageVersion.of(17)` → `of(25)`)

**Interfaces:**
- Consumes: the newest-stable Gradle wrapper from Task 1.
- Produces: a build that compiles and tests against Java 25, with the JDK auto-downloaded by Gradle when not present locally. Every later task relies on this toolchain (Docker and CI supply their own JDK 25, but must match the language version pinned here).

- [x] **Step 1: Add the foojay toolchain-resolver plugin to `settings.gradle`**

The build currently has no toolchain auto-provisioning and the local JDK is 21, so bumping the toolchain to 25 would otherwise fail with `No compatible toolchains found`. Add the Gradle-recommended resolver so Gradle can download JDK 25 on demand.

In `oag/settings.gradle`, add **above** the existing `rootProject.name` line:

```groovy
plugins {
    id 'org.gradle.toolchains.foojay-resolver-convention' version '0.8.0'
}

rootProject.name = 'oag'
```

If Gradle reports that `0.8.0` is not resolvable (some Gradle releases require a newer resolver), check https://plugins.gradle.org/plugin/org.gradle.toolchains.foojay-resolver-convention for the latest stable (ignore `-M`/`-RC`/`-alpha`/`-beta`) and use that instead.

- [x] **Step 2: Bump the toolchain language version**

In `oag/build.gradle`, change the `java { toolchain { … } }` block (line 37):

```groovy
java {
    toolchain {
        languageVersion = JavaLanguageVersion.of(25)
    }
}
```

(from `JavaLanguageVersion.of(17)`).

- [x] **Step 3: Verify Gradle resolves a Java 25 toolchain**

Run (from `oag/`): `./gradlew javaToolchains --console=plain`
Expected: the output lists a `Language Version 25` JDK as *detected* or *auto-provisioned* (foojay may download it on this first run — allow network access). If instead Gradle reports it does not support language version 25, the wrapper from Task 1 is too old — this should not happen with the latest GA; if it does, stop and report it (the latest stable Gradle is the ceiling this plan can use).

- [x] **Step 4: Full build on the Java 25 toolchain**

Run (from `oag/`): `./gradlew clean build --console=plain`
Expected: `BUILD SUCCESSFUL`, all 108 tests pass. This compiles every source file with `--release 25` semantics (Gradle applies the toolchain) and boots the full `@SpringBootTest` context on Java 25 — a passing run confirms routing, security, CSRF, session and OIDC behaviour survived the bump.

If compilation or a test fails:
- Read the failing symbol/test. Spring Boot 4.1 supports Java 25, so a failure is most likely a removed/deprecated-for-removal JDK API or a byte-manipulation/reflection library reacting to the newer class-file version. Apply the minimal source change in `oag/src/main` (or the dependency's documented replacement) and re-run.
- Treat any newly-failing integration test as a real behavioural regression to investigate, not a test to disable.

- [x] **Step 5: Commit**

```bash
git add oag/settings.gradle oag/build.gradle
git commit -m "build: move Java toolchain to 25 with foojay auto-provisioning"
```
